## Архитектура решения

**Цель**: дообучить BAAI/bge-m3 на геологическом корпусе, чтобы модель понимала специфическую терминологию (стратиграфические горизонты, типы пород, геохимические показатели, буровые параметры) и давала точный retrieval в RAG-пайплайне Open WebUI.

| Компонент | Инструмент | Назначение |
|-----------|-----------|-----------|
| **Базовая модель** | BAAI/bge-m3 | Мультиязычная (ru+en), dense vectors 1024-dim |
| **Фреймворк обучения** | SentenceTransformers ≥ 2.7 | Fine-tuning, evaluation, serving |
| **Сервинг** | FastAPI + uvicorn (06_serve.py) | OpenAI-compatible /v1/embeddings endpoint |
| **Подключение** | Open WebUI → RAG Settings → OpenAI Engine | HTTP API, без ollama |
| **Векторное хранилище** | ChromaDB (встроен в Open WebUI) | Автоматически используется после настройки |

## Подготовка данных

**Скрипты**: `01_chunk_documents.py` → `02_generate_queries.py` → `03_mine_hard_negatives.py`

### 1.1 Источники корпуса

- **Внутренние документы**: геологические отчёты, описания скважин, акты ГКЗ, технологические регламенты
- **Экспертные пары**: вопрос–чанк, аннотированные геологами (минимум 200–500 пар для dev/test)
- **Открытые источники**: публикации ВНИГРИ, ГКЗ, ФГБУ «Росгеология», научные статьи по петрографии, нефтегазу
- **Нормативная база**: классификаторы запасов, ГОСТы, методические указания Роснедра

**Геологические подобласти для автоматической разметки:**
- стратиграфия
- петрография
- тектоника
- геохимия
- гидрогеология
- рудная геология
- сейсмика
- ГИС/картография
- геофизика

### 1.2 Чанкование (01_chunk_documents.py)

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| **chunk_size** | 512 символов | Оптимально для bge-m3 (max 8192 токенов, но короткие чанки точнее) |
| **overlap** | 64 символа | Сохраняет контекст на границах (названия горизонтов, цифры) |
| **Мин. длина чанка** | 50 символов | Фильтрация заголовков и колонтитулов |
| **Метаданные чанка** | source, subdomain, chunk_index | Нужны для балансировки и фильтрации |

### 1.3 Генерация вопросов (02_generate_queries.py)

Для каждого чанка LLM генерирует **3–5 вопросов**, варьируя стиль от академического до производственного. Системный промпт настроен на геологическую экспертизу.

**Фильтрация вопросов** (аналог X-Intelligence §5):
- Удаляются тривиальные вопросы (ответ «да/нет», общие справочные вопросы)
- Удаляются невалидируемые вопросы (субъективные оценки)
- Требование: вопрос содержит геологическую специфику (термины, числа, названия)

### 1.4 Hard Negative Mining (03_mine_hard_negatives.py)

**Три стратегии из X-Intelligence 3.0 §3.1**, адаптированные под геологию:

#### BM25 Hard Negatives
- **Метод**: Лексически похожие чанки из другого документа (>70% слов пересекаются, но разный смысл)
- **Геологический пример**: 
  - ✓ «Пористость коллектора Тюменской свиты» 
  - ✓ «Пористость коллектора Васюганской свиты» 
  - → похожие слова, но разные объекты

#### Cross-domain Semantic
- **Метод**: Семантически похожие чанки из другой геологической подобласти (via dense retrieval)
- **Геологический пример**: 
  - Запрос про нефтяной коллектор 
  - → negative из рудной геологии (тоже про «вмещающие породы»)

#### Adversarial (LLM)
- **Метод**: Перефразировка с намеренным изменением ключевых фактов
- **Геологический пример**: 
  - Изменить «глубина 2450 м» → «глубина 3800 м»
  - «юрский возраст» → «меловой возраст»

**Periodic Hard Negative Mining** (аналог X-Intelligence §3.1):
- Каждые **1000 шагов обучения** текущая версия модели переиндексирует корпус
- Ищет новые трудные примеры — те чанки, которые модель уже начинает путать
- Это обновление помогает модели адаптироваться к своим ошибкам

### 1.5 Балансировка по подобластям

**Критично**: не более **20% триплетов от одной геологической подобласти** (subdomain distribution filter, X-Intelligence §2.2).

⚠️ Почему это важно: нефтегазовые документы обычно многочисленны и без балансировки «задавят» другие подобласти.

### 1.6 Целевые объёмы датасета

| Сплит | Минимум | Цель | Источник |
|-------|---------|------|----------|
| **Train (триплеты)** | 5 000 | 30 000–50 000 | Авто-генерация через LLM |
| **Dev (пары)** | 300 | 1 000 | Эксперт-аннотация |
| **Test (пары)** | 200 | 500 | Эксперт-аннотация (blind) |

## Эксперименты по дообучению

**Скрипт**: `04_train_embedder.py`

### 2.1 Функции потерь

| Loss | Класс SentenceTransformers | Вес | Назначение |
|------|-------------------------|-----|-----------|
| **MultipleNegativesRankingLoss** | `losses.MultipleNegativesRankingLoss` | Основной | In-batch negatives, масштабируется с batch_size. Аналог Contrast Loss из X-Intelligence §3.1 |
| **CoSENTLoss** | `losses.CoSENTLoss` | Дополнительный | Попарные scored примеры. Улучшает тонкую разметку близких смыслов |
| **TripletLoss** | `losses.TripletLoss` | Для явных hard negatives | Использует готовые триплеты (query, pos, hard_neg) |

#### Детальное описание лоссов

##### MultipleNegativesRankingLoss (MNRL)
- **Как работает**: берет query и его positive пример, затем все остальные примеры в батче становятся негативами (in-batch negatives)
- **Масштабируемость**: эффективнее работает с большим batch_size (32–64)
- **Идея**: контрастное обучение — query должен быть ближе к своему positive, чем к негативам
- **Применение**: быстрое обучение, базовая дифференциация семантики

##### CoSENTLoss
- **Как работает**: работает с попарными (pairwise) примерами с scored метками релевантности (не просто 0/1, а градации)
- **Идея**: позволяет модели различать степень семантической близости между примерами
- **Применение**: тонкая разметка, добавляет 3–7% улучшения к NDCG

##### TripletLoss
- **Как работает**: использует готовые триплеты **(query, positive, hard_negative)**
- **Идея**: query должен быть ближе к positive, чем к hard_negative, на заданное расстояние
- **Применение**: обучение на жестких примерах, особенно эффективен в E3–E4, когда добавлены adversarial и periodic negatives

### 2.2 Гиперпараметры

| Параметр | Значение |
|----------|----------|
| **Base model** | BAAI/bge-m3 |
| **Batch size** | 32 (16GB GPU) / 64 (40GB GPU) |
| **Learning rate** | 2e-5, cosine decay |
| **Warmup** | 10% от total steps |
| **Epochs** | 3–5 |
| **Mixed precision** | fp16 / bf16 |
| **Max seq length** | 512 токенов |
| **Normalize embeddings** | True (обязательно!) |

### 2.3 Матрица экспериментов

**Прогрессивная оптимизация** по аналогии с таблицей 9 (RAG components) из X-Intelligence 3.0:

| Экспл. | Изменение vs предыдущего | Ожидаемый эффект |
|--------|--------------------------|------------------|
| **E0 — Baseline** | BAAI/bge-m3 без дообучения | Стартовые метрики (NDCG@10 ~0.50–0.60) |
| **E1 — SFT+BM25** | Fine-tuning, только BM25 hard negatives | +5–10% NDCG@10 |
| **E2 — +Cross-domain** | Добавляем cross-domain semantic negatives | +3–7% (лучше граничные случаи между подобластями) |
| **E3 — +Adversarial** | Добавляем adversarial negatives | +2–5% (устойчивость к перефразировкам) |
| **E4 — +Periodic Mining** | Hard negative mining каждые 1000 шагов | +2–4% (финальный целевой результат) |

## Evaluation

**Скрипт**: `05_evaluate.py`

### 3.1 Подготовка к evaluation

Evaluation производится на **test set** — наборе пар `(query, relevant_document)`, которые модель НЕ видела при обучении (blind test).

**Наборы данных**:
- **Dev set**: ~1000 пар (промежуточная проверка во время обучения)
- **Test set**: ~500 пар (финальная оценка, blind — модель никогда не видела эти примеры)

⚠️ **Важно**: Test set должен быть полностью слепой для модели. Если модель видела test примеры при обучении, то NDCG будет 0.95+, но при реальном использовании упадет до 0.60 (переобучение).

### 3.2 Процесс evaluation

Для каждого `query` из test set:

```
1. Вычислить embedding query через дообученную модель
   embeddings_query = model.encode(query)

2. Вычислить embeddings всех документов в корпусе
   embeddings_corpus = [model.encode(doc) for doc in corpus]

3. Рассчитать косинусную близость между query и каждым документом
   similarities = cosine_similarity(embeddings_query, embeddings_corpus)
   scores = [0.92, 0.85, 0.81, 0.78, ..., 0.45]

4. Отсортировать документы по косинусной близости (по убыванию)
   ranked_docs = sorted_by_similarity(corpus)

5. Найти позицию relevant_document в отсортированном списке
   position = find_position(relevant_doc, ranked_docs)
```

**Пример конкретно:**

```
Query: "Какова пористость коллектора Тюменской свиты?"

Ранжирование документов:
  Позиция 1: Doc_A (0.92) ← ✅ RELEVANT (это наш positive)
  Позиция 2: Doc_B (0.85)
  Позиция 3: Doc_C (0.81)
  Позиция 4: Doc_D (0.78)
  ...
  Позиция 47: Doc_Z (0.45)

Результат: Relevant документ на позиции 1 → отлично!
```

### 3.3 Сбор статистики

После обработки всех query в test set собираются позиции relevant документов:

```
Query 1: relevant на позиции 1 ✅
Query 2: relevant на позиции 3 ✅
Query 3: relevant на позиции 8 ✅
Query 4: relevant НЕ в топ-10 ❌ (ошибка retrieval)
Query 5: relevant на позиции 2 ✅
...
Query 500: relevant на позиции 2 ✅
```

### 3.4 Метрики оценки

На основе собранных позиций считаются метрики:

#### NDCG@10 (Normalized Discounted Cumulative Gain)
- **Что означает**: на сколько хорошо релевантный документ ранжируется в топ-10
- **Как считается**: чем выше позиция релевантного документа, тем выше оценка (позиция 1 >> позиция 5)
- **Целевое значение**: **> 0.72**
- **Интерпретация**: 0.72 = модель кладет релевантный документ в топ по качеству примерно в 72% случаев

**Формула** (упрощённо):
```
NDCG@10 = (1 + log(position)) / log(10)
Позиция 1:  NDCG = 1.0
Позиция 2:  NDCG = 0.63
Позиция 3:  NDCG = 0.5
Позиция 5:  NDCG = 0.43
Позиция 10: NDCG = 0.3
```

#### Recall@5 и Recall@10
- **Recall@5**: доля запросов, где positive попал хотя бы в топ-5
  - Целевое: **> 0.75** (75% запросов находят ответ в топ-5)
- **Recall@10**: доля запросов, где positive в топ-10
  - Целевое: **> 0.85** (85% запросов находят ответ в топ-10)
- **Интерпретация**: высокий Recall = модель охватывает релевантный материал, но может ранжировать его не всегда в топе

**Пример**:
```
Test set: 500 запросов

Recall@5:  380/500 = 0.76 ✅
           (380 запросов имели relevant в топ-5)

Recall@10: 425/500 = 0.85 ✅
           (425 запросов имели relevant в топ-10)
```

#### MRR (Mean Reciprocal Rank)
- **Как считается**: позиция первого релевантного документа → берется 1/позиция → усредняется по всем запросам
- **Целевое значение**: **> 0.68**
- **Интерпретация**: в среднем релевантный документ находится на позиции ~1.5 (очень хорошо)

**Пример**:
```
Query 1: relevant на позиции 2  → 1/2 = 0.5
Query 2: relevant на позиции 1  → 1/1 = 1.0
Query 3: relevant на позиции 5  → 1/5 = 0.2
...
Query 500: relevant на позиции 2 → 1/2 = 0.5

MRR = (0.5 + 1.0 + 0.2 + ... + 0.5) / 500 = 0.68 ✅
```

#### ARR@5 (Acceptable Retrieval Rate)
- **Что означает**: доля запросов с хотя бы одним релевантным документом в топ-5
- **Целевое значение**: **> 0.80** (80% запросов имеют хотя бы один релевантный результат в топ-5)
- **Практическое значение**: показатель "пользователь найдет ответ, прокрутив первые 5 результатов"

**Пример**:
```
Test set: 500 запросов
400/500 = 0.80 ✅ (в 400 запросах был хотя бы 1 релевант в топ-5)
```

### 3.6 Прогресс по экспериментам

Пример типичного улучшения метрик через E0→E4:

```
E0 (baseline):     NDCG = 0.52  (модель не обучена)
E1 (SFT+BM25):     NDCG = 0.58  (+0.06) ✅
E2 (+Cross-dom):   NDCG = 0.63  (+0.05) ✅

Recall@5: 0.55 → 0.76 (+0.21)
MRR:      0.48 → 0.69 (+0.21)
```

### 3.7 Связь evaluation с обучением

**Во время обучения**:
- Dev set используется для early stopping (если NDCG на dev не улучшается 2+ эпохи → стоп)
- Проверка каждые 500 шагов

**После обучения**:
- Test set используется для финальной оценки (видит только финальную модель)
- Результаты test set сравниваются со целевыми значениями

## Риски и митигации

| Риск | Митигация |
|------|-----------|
| **Мало геологических документов (<500 страниц)** | Синтетическая аугментация: генерировать дополнительные пары через LLM из имеющихся чанков. Минимум 5 000 триплетов достаточно для заметного улучшения. |
| **Catastrophic forgetting (ухудшение общих возможностей)** | Добавить 10–20% общедоменных пар в обучение (Math/Science из открытых датасетов) |
| **Доминирование нефтегазовой подобласти** | Обязательная балансировка: не >20% на подобласть через subdomain distribution filter |
| **Медленный inference на CPU** | Квантование через optimum: `optimum-cli export onnx --model ./finetuned-bge-m3-geo` |
| **Нарушение blind test** | Проверить, что test set полностью отделен от train/dev до начала обучения |
| **Переобучение на hard negatives** | Periodical mining каждые 1000 шагов обновляет трудные примеры под текущую версию модели |
| **Несогласованность метрик** | Использовать одну и ту же eval процедуру для всех экспериментов E0–E4 |
